Add-Customer 表单优化

May 22, 2018

在测试填写这个表单时，发现有下面几个问题：

  1. 联系方式填写不合理
  2. 相同特征的信息没有放在一起
  3. 信息的阅读性混乱，文案长短相差有点大
  4. 校验出错提示不能明确告诉使用者如何解决，反而还容易造成误解
  5. 表单区域的空间利用的太拥挤

1. 联系方式的定义优化

字段定义:

Mobile Number ：只能填写手机号码；Secondary Number ：填写其它联系号码，包括手机和固话

  1. 限制死了只能填手机号码、而另一个可以填固话或手机号；

  2. 不同国家的手机号码位数不同，美国10位、加拿大10位、香港8位、新加坡8位，所以不能像国内一样可以按11位数字格式判断这是否是个手机号码；

  3. 其次是国外的手机号码可能会与固话是一样的位数，比如加拿大，手机和固话都是10位数，类似格式647-348-XXXX，有时他们自己也都不分清这是手机号码还是固话；

  4. 隐私问题，产品的字段定义这里必须要填写手机，但毕竟国外对个人隐私比较注重，手机号并不是所有人都愿意提供给第三方的，接收自己并不想要的广告或造成信息泄露；

所以，Mobile Number这里并不能判断填写的是否是真实的手机号码，也不能强制要求填写手机号码。

首先想想为什么要填写联系号码，对于提供服务的店铺来说，获取顾客的联系号码对他们有以下的作用：

  1. 联系客服确认预约；
  2. 顾客回访，了解服务效果；
  3. 给重要顾客提供优惠信息以及关系维护；

而这些，邮件（国外有使用邮件交流的习惯）同样是可以达到目的，但相比打个电话就缺乏时效性和关怀性。所以联系号码有它的存在意义，但并不是必要的。而且鉴于隐私原因，顾客的个人手机也并不是必须要填写。

根据这些，把Mobile Number 和Secondary Number 是可以统一整合成一项联系号码的录入，分为主要号码和次要号码，不需要像之前一样做这么明显的手机与固话划分。如下图

2. 信息关联性和字段简化

  1. 在这个表单中，2（联系号码）和4（邮箱）都是属于个人的联系方式，根据格式塔的相似性原理，这两个具有明显共同特性的信息应该放在一起，不应该像现在一样在它们中间插入了一个Birthday。

  2. 在不影响信息清晰传达的前提下，尽量减少不需要的文字。

    图中的 Mobile Number 、Secondary Number、Email Address、 Customer Note，文案过于累赘，简化不必要的单词，减少阅读的信息量，并且表单整体左右视觉可于拥挤，适当优化间距，提高信息的阅读性。

3. 校验问题

由于并不是所有顾客都愿意提供姓名、联系号码、邮箱，因此没有固定某一项内容是必填，而把这几项能识别顾客的信息只需要其中一个即可。

添加顾客的时候Name、Phone、Email中共5个输入框，至少要填写任意一项，不然在创建或修改顾客时，无法知道填写的是哪一个顾客。

但现在校验提醒的时候只针对了联系方式，这里就会容易让人误解成 Phone Number是必填项。

如果针对其中某一项进行提示会让人产生误解只有该项必填，并且提示的内容与该项的信息并不完全等同更会让人费解，这里可以使用非浮层alert提示。
